# PostgreSQL Setup And Data Migration

This document describes how to:

- configure this project to use a local PostgreSQL database
- create the required tables
- run Django migrations
- sync building metadata from `assets/` into the `buildings` table
- migrate existing collected activity data from the current SQLite database into PostgreSQL

## Overview

The app uses two kinds of database tables:

1. Django-managed tables created by `python manage.py migrate`
2. One existing unmanaged table, `buildings`, which must already exist in PostgreSQL

The `buildings` table is not created by Django migrations. Instead:

- Django reads it through the unmanaged `Building` model
- after every `manage.py migrate`, the app scans `assets/`
- discovered buildings and floors are inserted or updated in `buildings`

## Required Tables

If you keep the current `INSTALLED_APPS`, these tables are expected in PostgreSQL:

### Manually created or pre-existing

- `buildings`

Current expected schema:

```sql
CREATE TABLE IF NOT EXISTS buildings (
  id integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  name text NOT NULL,
  address text NOT NULL DEFAULT '',
  floors text NOT NULL,
  created_at timestamp without time zone NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

Recommended:

```sql
CREATE UNIQUE INDEX IF NOT EXISTS buildings_name_unique_idx
ON buildings (name);
```

Notes:

- `name` stores the building id from `assets/`, for example `SUTD` or `One-Punggol`
- `floors` stores JSON text, not a relational child table
- non-empty `address` values are preserved by the sync process

## 1. Create PostgreSQL User And Database

Example using `psql` as a PostgreSQL admin user:

```sql
CREATE ROLE indoor_app WITH LOGIN PASSWORD 'change_this_password';
CREATE DATABASE indoor_activities OWNER indoor_app;
```

If the database already exists, reuse it.

Then connect to it:

```bash
psql -U indoor_app -d indoor_activities -h 127.0.0.1
```

## 2. Create Or Verify The `buildings` Table

If your database already has this table, verify that it matches:

- `id` integer primary key
- `name` text
- `address` text
- `floors` text
- `created_at` timestamp without time zone

If the table does not exist yet, create it:

```sql
CREATE TABLE IF NOT EXISTS buildings (
  id integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  name text NOT NULL,
  address text NOT NULL DEFAULT '',
  floors text NOT NULL,
  created_at timestamp without time zone NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE UNIQUE INDEX IF NOT EXISTS buildings_name_unique_idx
ON buildings (name);
```

## 3. Configure Local Credentials Safely

Create a `.env` file in the project root.

This file is git-ignored and should not be committed.

```dotenv
DJANGO_DB_ENGINE=postgresql
DJANGO_DB_NAME=indoor_activities
DJANGO_DB_USER=indoor_app
DJANGO_DB_PASSWORD=change_this_password
DJANGO_DB_HOST=127.0.0.1
DJANGO_DB_PORT=5432
```

Load it before running Django commands:

```bash
set -a
source .env
set +a
```

## 4. Install Python Dependencies

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

The project now needs:

- `Django`
- `psycopg[binary]`

## 5. Run Migrations

With the PostgreSQL environment loaded:

```bash
./.venv/bin/python manage.py migrate
```

What this does:

- creates all Django-managed tables
- applies app schema changes for `collector_activityrecord`
- triggers the post-migrate building sync
- reads building/floor data from `assets/`
- inserts or updates rows in `buildings`

If the `buildings` table does not exist, the sync is skipped safely.

## 6. Verify The Database

In `psql`:

```sql
\dt
SELECT COUNT(*) FROM buildings;
SELECT COUNT(*) FROM collector_activityrecord;
```

You should see at least:

- `buildings`
- `collector_activityrecord`
- the Django framework tables listed above

To inspect synced buildings:

```sql
SELECT id, name, address, floors
FROM buildings
ORDER BY id;
```

The `floors` column should look like JSON text similar to:

```json
[{"id":"ground-floor","label":"Ground Floor","mapSrc":"/assets/SUTD/ground-floor.svg"}]
```

## 7. Migrate Existing Collected Data From SQLite To PostgreSQL

The current local project database is `db.sqlite3`.

The safest migration path is:

1. export only the app data from SQLite
2. switch Django to PostgreSQL
3. load the exported data into PostgreSQL

### Step 7.1 Export activity records from SQLite

Run this without PostgreSQL env vars loaded, so Django uses the local SQLite database:

```bash
unset DJANGO_DB_ENGINE DJANGO_DB_NAME DJANGO_DB_USER DJANGO_DB_PASSWORD DJANGO_DB_HOST DJANGO_DB_PORT
./.venv/bin/python manage.py dumpdata collector.ActivityRecord --indent 2 > activity_records.json
```

This exports only the collected activity records.

### Step 7.2 Switch back to PostgreSQL

```bash
set -a
source .env
set +a
```

### Step 7.3 Ensure PostgreSQL schema exists

```bash
./.venv/bin/python manage.py migrate
```

### Step 7.4 Load the exported records

```bash
./.venv/bin/python manage.py loaddata activity_records.json
```

### Step 7.5 Verify row counts

```sql
SELECT COUNT(*) FROM collector_activityrecord;
```

## 8. Building Sync Behavior

On every `manage.py migrate`:

- the app scans `assets/`
- each building folder becomes one row in `buildings`
- floors are serialized into the `floors` text column as JSON
- rows are matched by `name`
- existing `floors` values are refreshed
- existing non-empty `address` values are preserved
- the synthetic shared root entry `__root__` is not inserted into `buildings`

Examples from the current `assets/` folder:

- `Heartbeat@Bedok`
- `One-Punggol`
- `SUTD`

## Troubleshooting

### `ModuleNotFoundError: No module named 'psycopg'`

Install dependencies again:

```bash
pip install -r requirements.txt
```

### `buildings` remains empty after migrate

Check:

- the `buildings` table exists in PostgreSQL
- your PostgreSQL credentials are loaded in the shell
- your `assets/` folder contains building directories with supported map files

Then rerun:

```bash
./.venv/bin/python manage.py migrate
```

### Permission errors in PostgreSQL

Ensure the app user can:

- connect to the database
- read and write tables in the `public` schema
- create tables during `migrate`

### Do not store DB credentials in frontend files

Never place database credentials in:

- `app.js`
- `collector/static/collector/app.js`
- HTML templates

Only the Django backend should connect to PostgreSQL.

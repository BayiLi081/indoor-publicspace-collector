# PostgreSQL Setup

This guide explains how to connect this Django app to PostgreSQL for deployment.

It is written for the current project setup:

- Django app deployed to a cloud host such as Azure App Service
- PostgreSQL used as the application database
- one unmanaged `buildings` table that must exist before first migration
- Django-managed tables created by `manage.py migrate`
- automatic building sync from `assets/` after migration

## How This App Uses PostgreSQL

This project reads database settings from environment variables. When `DJANGO_DB_ENGINE=postgresql`, Django connects to PostgreSQL instead of SQLite.

This app uses two categories of tables:

1. Django-managed tables created by migrations
2. one manual table, `buildings`, which Django does not create

Important:
The `buildings` table must exist before you run `python manage.py migrate` in the deployed environment.

## Prerequisites

Before continuing, make sure you already have:

- a PostgreSQL server
- a database for this app, for example `indoor_activities`
- a PostgreSQL user for this app, for example `indoor_app`
- network access from the Django app host to PostgreSQL
- SSL/TLS requirements confirmed with your provider

## Recommended Setup Order

1. Create the PostgreSQL database and app user.
2. Create the `buildings` table.
3. Add the PostgreSQL environment variables in the app host.
4. Deploy the Django app.
5. Run `python manage.py migrate`.
6. Verify that Django tables and `buildings` data are present.

## Step 1. Create the PostgreSQL Database and User

Example using `psql` as an admin user:

```sql
CREATE ROLE indoor_app WITH LOGIN PASSWORD 'change_this_password';
CREATE DATABASE indoor_activities OWNER indoor_app;
```

If your cloud provider already created the database, reuse it and create only the application user if needed.

## Step 2. Create the `buildings` Table

This table is required by the current project and is not created by Django migrations.

Run:

```sql
CREATE TABLE IF NOT EXISTS buildings (
  id integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  name text NOT NULL,
  address text NOT NULL DEFAULT '',
  floors text NOT NULL,
  created_at timestamp without time zone NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE UNIQUE INDEX IF NOT EXISTS buildings_name_unique_idx
ON buildings (name);
```

Notes:

- `name` stores the building id discovered from `assets/`
- `floors` stores JSON text
- non-empty `address` values are preserved during building sync

## Step 3. Configure the App Environment Variables

Add these settings in your deployment environment.

Example for Azure App Service:

1. Open the App Service.
2. Go to `Settings` -> `Environment variables`.
3. Add the values below.
4. Save the changes.

Base Django and PostgreSQL settings:

```text
DJANGO_DEBUG=False
DJANGO_ALLOWED_HOSTS=<your-app>.azurewebsites.net,<your-custom-domain>
DJANGO_CSRF_TRUSTED_ORIGINS=https://<your-app>.azurewebsites.net,https://<your-custom-domain>

DJANGO_DB_ENGINE=postgresql
DJANGO_DB_NAME=indoor_activities
DJANGO_DB_USER=indoor_app
DJANGO_DB_PASSWORD=<strong-password>
DJANGO_DB_HOST=<your-postgres-host>
DJANGO_DB_PORT=5432
DJANGO_DB_SSLMODE=require

DJANGO_SYNC_BUILDINGS_ON_MIGRATE=True
DJANGO_SESSION_COOKIE_SECURE=True
DJANGO_CSRF_COOKIE_SECURE=True
```

Optional database settings supported by this project:

```text
DJANGO_DB_SSLROOTCERT=/path/to/ca.pem
DJANGO_DB_CONNECT_TIMEOUT=30
```

Use these when required by your PostgreSQL provider.

Important:
Store these values in the app host configuration or secret manager. Do not commit database passwords to the repository.

## Step 4. Deploy the App

Deploy the Django app using your normal deployment path.

For Azure App Service, that typically means:

- GitHub Actions deployment
- ZIP deploy
- App Service deployment center

Before running migrations, make sure the deployed app has:

- access to the PostgreSQL host
- the PostgreSQL environment variables above
- Python dependencies installed from `requirements.txt`

## Step 5. Run Migrations

After deployment, run:

```bash
python manage.py migrate
```

This step will:

- create Django-managed tables
- apply schema changes for app models
- trigger the post-migrate building sync
- scan `assets/`
- insert or update building rows in `buildings`

If `buildings` does not exist, the sync is skipped safely, but the app will not have the expected building metadata.

## Step 6. Verify the Database

After migration, verify the following:

- the app starts successfully
- the `buildings` table contains rows
- Django tables were created
- the app can save and read records

Useful SQL checks:

```sql
SELECT COUNT(*) FROM buildings;
SELECT COUNT(*) FROM collector_activityrecord;
SELECT id, name FROM buildings ORDER BY id;
```

Expected Django-managed tables include:

- `collector_activityrecord`
- `collector_siteobservation`
- `django_migrations`
- `django_content_type`
- `auth_permission`
- `auth_group`
- `auth_group_permissions`
- `auth_user`
- `auth_user_groups`
- `auth_user_user_permissions`
- `django_admin_log`
- `django_session`

## What Happens During `migrate`

When you run `python manage.py migrate`, this project does more than standard Django schema setup.

It also:

- scans the `assets/` directory
- converts building folders into rows in `buildings`
- serializes floor metadata into the `floors` text column as JSON
- preserves existing non-empty `address` values
- skips the synthetic shared root entry `__root__`

## Azure Example

If you are deploying on Azure, this is the usual flow.

### 1. Create Azure Database for PostgreSQL

Create:

- Azure Database for PostgreSQL Flexible Server
- database: `indoor_activities`
- application user: `indoor_app`

Recommended:

- restrict public access where possible
- allow only the app environment or approved client IPs
- require SSL/TLS
- enable automated backups

### 2. Add App Service Settings

Set the PostgreSQL settings shown above in App Service `Environment variables`.

### 3. Ensure Network Access Works

Confirm:

- the PostgreSQL firewall allows the app host
- private networking is configured if you use it
- DNS resolution and SSL requirements are correct

### 4. Run `migrate`

Run:

```bash
python manage.py migrate
```

Then verify the site can load building data and save records.

## Common Problems

### The app cannot connect to PostgreSQL

Usually caused by:

- wrong host, username, password, or port
- firewall or network restrictions
- SSL required but not configured
- wrong database name

Check:

- `DJANGO_DB_HOST`
- `DJANGO_DB_PORT`
- `DJANGO_DB_NAME`
- `DJANGO_DB_USER`
- `DJANGO_DB_PASSWORD`
- `DJANGO_DB_SSLMODE`

### The app starts, but `buildings` stays empty

Usually caused by:

- `buildings` table missing
- `python manage.py migrate` not run after deployment
- `DJANGO_SYNC_BUILDINGS_ON_MIGRATE=False`
- missing or empty `assets/` content in the deployed app

### Migrations fail with permission errors

The app user typically needs permission to:

- connect to the database
- use the target schema
- create tables during migration
- read and write project tables

### `403 Forbidden` after deployment

This is not a PostgreSQL issue. Usually it means:

- `DJANGO_ALLOWED_HOSTS` is missing or incorrect
- `DJANGO_CSRF_TRUSTED_ORIGINS` is missing or incorrect

## Operational Recommendations

- Use a dedicated application user, not the database admin user.
- Keep `DJANGO_DEBUG=False` in deployment.
- Require SSL/TLS for production connections.
- Rotate database passwords periodically.
- Store secrets in the cloud platform secret store or app configuration.
- Turn on automated backups.
- Restrict inbound access as much as possible.

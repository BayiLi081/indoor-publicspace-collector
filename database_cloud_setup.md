# Cloud Database Deployment Guide

This guide explains how to deploy the database side of this project to the cloud and connect the Django app to it safely.

It is written for the current project setup:

- Django web app
- PostgreSQL as the production database
- one unmanaged `buildings` table
- Django-managed app/framework tables
- automatic building sync from `assets/` after `manage.py migrate`

This guide uses Azure terminology where helpful because the repo already includes Azure App Service deployment workflow, but the same structure works for other cloud providers with managed PostgreSQL.

## Recommended Cloud Architecture

Use:

- one managed PostgreSQL instance
- one dedicated database, for example `indoor_activities`
- one dedicated application user, for example `indoor_app`
- one Django app deployment target, for example Azure App Service
- cloud-managed secrets or app settings, not hardcoded credentials

Recommended separation:

- application code deployed from GitHub
- database hosted on a managed PostgreSQL service
- database credentials stored in cloud secret storage or application settings

## Required Database Objects

There are two categories of tables:

### 1. Pre-existing manual table

You must create this table yourself before first production migration:

- `buildings`

Expected schema:

```sql
CREATE TABLE IF NOT EXISTS buildings (
  id integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  name text NOT NULL,
  address text NOT NULL DEFAULT '',
  floors text NOT NULL,
  created_at timestamp without time zone NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE UNIQUE INDEX IF NOT EXISTS buildings_name_unique_idx
ON buildings (name);
```

### 2. Django-managed tables

Created by `python manage.py migrate`:

- `collector_activityrecord`
- `django_migrations`
- `django_content_type`
- `auth_permission`
- `auth_group`
- `auth_group_permissions`
- `auth_user`
- `auth_user_groups`
- `auth_user_user_permissions`
- `django_admin_log`
- `django_session`

For the full local database setup and schema notes, see [POSTGRESQL_SETUP.md](./POSTGRESQL_SETUP.md).

## What Happens During `migrate`

When you run:

```bash
python manage.py migrate
```

the app will:

- create Django-managed tables
- apply the `collector_activityrecord` schema
- trigger the `post_migrate` signal
- scan `assets/`
- upsert buildings and floor metadata into `buildings`

If `buildings` does not exist, the sync is skipped safely.

## Cloud Provider Checklist

For any provider, you need:

1. a PostgreSQL server
2. a database named `indoor_activities` or equivalent
3. an application user with login permissions
4. network access from the app to the database
5. SSL/TLS enabled if required by the provider
6. environment variables configured in the app host
7. `buildings` table created before first `migrate`

## Azure Example

The repo already contains an Azure App Service GitHub Actions workflow, so Azure is the most natural cloud path for this project.

### Step 1. Create Azure Database For PostgreSQL

Create:

- Azure Database for PostgreSQL Flexible Server
- database: `indoor_activities`
- application user: `indoor_app`

Recommended:

- disable public exposure unless needed
- restrict firewall/network rules to the app environment
- require SSL/TLS connections
- enable automated backups

### Step 2. Create The `buildings` Table

Connect with `psql`, Azure Data Studio, or the Azure portal query tool and run:

```sql
CREATE TABLE IF NOT EXISTS buildings (
  id integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  name text NOT NULL,
  address text NOT NULL DEFAULT '',
  floors text NOT NULL,
  created_at timestamp without time zone NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE UNIQUE INDEX IF NOT EXISTS buildings_name_unique_idx
ON buildings (name);
```

### Step 3. Configure App Settings In Azure App Service

Set these application settings in the web app:

```text
DJANGO_DEBUG=False
DJANGO_ALLOWED_HOSTS=<your-app>.azurewebsites.net,<your-custom-domain>
DJANGO_CSRF_TRUSTED_ORIGINS=https://<your-app>.azurewebsites.net,https://<your-custom-domain>

DJANGO_DB_ENGINE=postgresql
DJANGO_DB_NAME=indoor_activities
DJANGO_DB_USER=indoor_app
DJANGO_DB_PASSWORD=<strong-password>
DJANGO_DB_HOST=<your-postgres-host>
DJANGO_DB_PORT=5432
DJANGO_DB_SSLMODE=require

DJANGO_SYNC_BUILDINGS_ON_MIGRATE=True
DJANGO_SESSION_COOKIE_SECURE=True
DJANGO_CSRF_COOKIE_SECURE=True
```

Optional if your provider requires a CA certificate:

```text
DJANGO_DB_SSLROOTCERT=/path/to/ca.pem
```

The project now supports these optional database connection settings:

- `DJANGO_DB_SSLMODE`
- `DJANGO_DB_SSLROOTCERT`
- `DJANGO_DB_CONNECT_TIMEOUT`

### Step 4. Deploy The App

Use the existing GitHub Actions workflow or your normal App Service deployment path.

Important:

- do not store database passwords in the repo
- put secrets in Azure App Service configuration or GitHub Secrets
- do not put DB credentials in frontend files or templates

### Step 5. Run Migrations Against The Cloud Database

After the app settings are configured, run:

```bash
python manage.py migrate
```

This can be done:

- from a deployment script
- from an App Service SSH session
- from a CI/CD job after deployment

This step is required to:

- create Django tables
- apply schema changes
- populate `buildings` from `assets/`

## Other Cloud Providers

The same structure works on:

- AWS RDS for PostgreSQL
- Google Cloud SQL for PostgreSQL
- Render PostgreSQL
- Railway PostgreSQL
- Supabase PostgreSQL

You only need to map their connection details into:

```text
DJANGO_DB_ENGINE=postgresql
DJANGO_DB_NAME=...
DJANGO_DB_USER=...
DJANGO_DB_PASSWORD=...
DJANGO_DB_HOST=...
DJANGO_DB_PORT=5432
DJANGO_DB_SSLMODE=require
```

## Secrets Management

Safe places for credentials:

- Azure App Service Application Settings
- Azure Key Vault
- GitHub Actions Secrets
- provider-native secret managers

Unsafe places:

- `.js` files
- HTML templates
- committed `.env` files
- committed Python source with plaintext passwords

## Data Migration To Cloud PostgreSQL

If you already collected data locally in SQLite:

1. export SQLite records
2. point Django to cloud PostgreSQL
3. run `migrate`
4. load the exported data

## Verification Checklist

After deployment, verify:

- the app starts successfully
- `GET /api/buildings/` returns expected buildings/floors
- the `buildings` table contains rows from `assets/`
- new records are saved into `collector_activityrecord`
- delete and export actions work
- CSRF errors are not occurring on the deployed domain

Useful checks in PostgreSQL:

```sql
SELECT COUNT(*) FROM buildings;
SELECT COUNT(*) FROM collector_activityrecord;
SELECT id, name FROM buildings ORDER BY id;
```

## Operational Recommendations

- Use a dedicated DB user, not the PostgreSQL admin user.
- Turn on automated backups.
- Restrict inbound network rules.
- Prefer private networking between app and database when possible.
- Require SSL/TLS for production DB connections.
- Keep `DJANGO_DEBUG=False` in cloud.
- Rotate DB passwords periodically.

## Common Failure Cases

### `403 Forbidden` in production

Usually caused by missing:

- `DJANGO_ALLOWED_HOSTS`
- `DJANGO_CSRF_TRUSTED_ORIGINS`

### Database connection failure

Usually caused by:

- wrong host or password
- firewall/network restrictions
- missing SSL requirement
- missing `psycopg` dependency

### `buildings` table not populated

Usually caused by:

- `buildings` table missing
- `manage.py migrate` not run after deployment
- `DJANGO_SYNC_BUILDINGS_ON_MIGRATE=False`
- empty or invalid `assets/` deployment content

### App can connect but cannot write

Usually caused by insufficient DB permissions for the application user.

Grant the app user permission to:

- connect to the database
- use the schema
- create tables during migration
- read and write project tables
